Camada 1

A intuição inicial

Imagine uma pilha de bandejas de restaurante self-service: bandejas novas vão em cima das que já estavam ali; quem precisa de uma pega a do topo, nunca a do meio nem a do fundo.

Toda a ideia é essa: "só vejo e só mexo no topo" — a última coisa a entrar é a primeira a sair, e o que está embaixo só fica acessível depois que tudo em cima foi retirado.

Essa restrição extremamente simples define a estrutura chamada Pilha.

Camada 2

Definição informal — vocabulário básico

Uma Pilha (stack) é uma estrutura linear de elementos com uma restrição de acesso clara:

Insere-se e remove-se sempre na mesma extremidade, chamada topo (top).

Os elementos abaixo do topo ficam ocultos — não há operação que os acesse diretamente. A política recebe o nome LIFO (Last-In, First-Out).

Tratamento canônico: Tenenbaum, cap. 2 — Pilhas; Sedgewick, Parte 1, cap. 4 — pushdown stacks.

As operações

  • empilhar (push) — coloca um novo elemento no topo.
  • desempilhar (pop) — remove e devolve o elemento do topo.
  • topo — apenas lê o elemento do topo, sem removê-lo.
  • vazia — informa se há ou não elementos na pilha.
Como na Fila, a restrição é a feature, não a limitação — é o que torna a Pilha simples, previsível e rápida.

Camada 3 — ponte com a Aula 02

O nó da Pilha é o nó da lista simples

A representação interna desta aula é construída sobre o nó apresentado em "O nó — unidade de construção" na Aula 02.

  • Mesmo nó: campo valor (carga útil) + campo proximo (ligação).
  • O que muda: política LIFO (em vez de FIFO) e estado externo enxuto — um único ponteiro topo, sem fim.

Camada 3

Um único ponteiro torna tudo simples

  • Fila precisa de dois ponteiros (inicio + fim) para que ambas as operações sejam O(1).
  • Pilha precisa de um só: o ponteiro topo. Empilhar = inserir no início. Desempilhar = remover do início.
Consequência didática: o código da Pilha não tem casos especiais. desempilhar faz topo = topo->proximo e pronto — se virar NULL, a pilha ficou vazia. Sem armadilha.

As operações em detalhe

  • empilhar(x) — aloca novo nó; novo->proximo = topo; topo = novo. O(1).
  • desempilhar() — lê valor de topo; salva topo->proximo em topo; libera o nó antigo. O(1).
  • topo() — devolve topo->valor sem alterar. O(1).
  • vazia() — testa topo == NULL. O(1).

Atenção em desempilhar: primeiro atualizar topo, só depois chamar free — inverter cria ponteiro suspenso (dangling pointer).

Invariantes — sempre verdadeiras

  • Pilha vazia: topo == NULL.
  • Pilha com pelo menos um elemento: topo != NULL; seguindo proximo a partir de topo chega-se a um nó cujo proximo é NULL (o fundo).
Pilha tem menos invariantes que a Fila porque tem menos estado — um ponteiro em vez de dois.

Camada 4

TAD Pilha — assinaturas

criar()                  -> Pilha
empilhar(Pilha, Inteiro) -> Pilha
desempilhar(Pilha)       -> Pilha       [erro se vazia]
topo(Pilha)              -> Inteiro     [erro se vazia]
vazia(Pilha)             -> Booleano

O TAD define a estrutura pelo seu contrato observável, sem fixar representação interna (Tenenbaum, cap. 2).

Axiomas — coração do LIFO

  • A1. vazia(criar()) = verdadeiro
  • A2. vazia(empilhar(P, x)) = falso
  • A3. topo(empilhar(P, x)) = x
  • A4. desempilhar(empilhar(P, x)) = P
    → LIFO: empilhar e desempilhar devolve a pilha original.
  • A5. topo(criar()) = erro
  • A6. desempilhar(criar()) = erro
Compare com a Fila: desenfileirar(enfileirar(F, x)) = enfileirar(desenfileirar(F), x) — lá o x "passa por dentro" da remoção. Aqui, ele simplesmente sai.
Eddy

Turma, segundo Tenenbaum, é justamente o axioma desempilhar(empilhar(P, x)) = P que define o comportamento LIFO de forma puramente algébrica, sem precisar falar em ponteiros ou memória. Tenenbaum, cap. 2 — Pilhas

Representação interna como tupla

Para a versão simplesmente encadeada com ponteiro topo:

P = (N, valor, proximo, topo)
  • N — conjunto (possivelmente vazio) dos nós da pilha.
  • valor: N → V — associa cada nó a um inteiro.
  • proximo: N → N ∪ {NULL} — associa cada nó ao seguinte (mais perto do fundo), ou NULL se for o fundo.
  • topo ∈ N ∪ {NULL} — ponteiro externo para o último elemento empilhado.

Invariante: vazia ⇔ topo = NULL.

Camada 5

Análise — operações principais

Análise canônica: CLRS, cap. 10.1; Tenenbaum, cap. 2.

Operação Vetor com topo-índice Lista encadeada com topo
empilharO(1)*O(1)
desempilharO(1)O(1)
topoO(1)O(1)
vaziaO(1)O(1)

* Vetor pode precisar de realloc eventual O(n) — distribuído, custo amortizado O(1).

Vetor × lista — escolha de engenharia

Vetor

  • Uma única alocação.
  • Mais econômico em memória.
  • Acesso muito rápido.
  • Capacidade fixa ou paga realloc.

Lista encadeada

  • Cresce sob demanda.
  • Capacidade ilimitada.
  • Código sem casos especiais.
  • 8 B/nó + 1 malloc por elemento.

Esta aula implementa a versão encadeada — mais simples, tamanho dinâmico. O custo de ponteiros é o "preço do ponteiro" da Aula 02.

Camada 6

Onde a Pilha aparece

  • Pilha de chamadas — toda função empilha um frame; return desempilha.
  • Ctrl+Z — cada ação executada vira entrada empilhada; cada desfazer desempilha.
  • "Voltar" do navegador — cada página visitada empilha; voltar desempilha.
  • Validação de parênteses — abre empilha; fecha desempilha e confere.
  • Recursão → iteração — pilha explícita substitui a pilha implícita do programa.

Variantes da Pilha

  • Vetor com topo-índice — com ou sem realloc.
  • Lista simplesmente encadeada com topo — a desta aula.
Lista dupla e circular não trazem ganho — todas as operações já estão em O(1) com lista simples; ponteiros extras só acrescentam custo de memória.

Pilha aparece como peça de construção em algoritmos clássicos — DFS (busca em profundidade) em grafos, em aulas futuras.

Bloco 3

Por que Pilha existe?

Como o Ctrl+Z funciona?

Ctrl+Z desfaz a última ação. De novo, a penúltima. De novo, a antepenúltima.

Como o editor sabe? Mantém uma Pilha de ações:

  • Cada ação do usuário → empilhar.
  • Cada Ctrl+Z → desempilhar e reverter.
  • Ctrl+Y (refazer) → segunda pilha que recebe o desfeito.
Uma Fila aqui seria desastrosa: começaria desfazendo a primeira ação ("abrir o arquivo"), não a mais recente.

A pilha de chamadas

Função A chama B, que chama C: 

[ chamada de C ]   <- topo (executando)
[ chamada de B ]
[ chamada de A ]
[ ..... main ..... ]

Cada return desempilha e devolve o controle a quem chamou.

É literalmente uma Pilha — o nome não é coincidência.

Eddy

Turma, a stack overflow que vocês já viram em mensagens de erro é exatamente essa pilha de chamadas estourando: cada chamada empilha um frame, e quando há recursão sem caso base a pilha cresce sem parar até esgotar a região de memória reservada para ela.

Bloco 4

Analogias

Pilha de bandejas no restaurante self-service

Bandejas chegam limpas → vão em cima.
Você precisa de uma → pega a do topo.

A pilha cresce por cima quando vêm novas; encolhe pelo topo quando saem. Você nunca vê a do meio.

LIFO literal.

Pilha de pratos sujos

Você termina a refeição → o prato vai em cima.

Quando vai lavar → começa pelo topo (o último colocado).

Última coisa suja, primeira coisa limpa.

Bloco 5

Código em C

Lista simplesmente encadeada com topo apenas.

Um único arquivo: pilha.c

Tudo num lugar só: #includes, structs, funções e main().

Em uma aula futura sobre organização de projetos em C, veremos como dividir uma TAD em pilha.h + pilha.c (separação interface / implementação). Por enquanto, manter tudo num arquivo facilita ler de cima a baixo.

pilha.c — includes e structs

#include <stdio.h>
#include <stdlib.h>

// Cada elemento da pilha vira um No.
typedef struct No {
    int valor;
    struct No* proximo;
} No;

// A pilha guarda apenas o topo.
typedef struct Pilha {
    No* topo;
} Pilha;

Pilha* pilha_criar(void) {
    Pilha* p = malloc(sizeof(Pilha));
    if (p == NULL) {
        printf("erro: memoria insuficiente\n");
        exit(1);
    }
    p->topo = NULL;
    return p;
}

pilha_empilhar

void pilha_empilhar(Pilha* p, int valor) {
    No* novo = malloc(sizeof(No));
    if (novo == NULL) {
        printf("erro: memoria insuficiente\n");
        exit(1);
    }
    novo->valor = valor;

    // O proximo do novo no e' o antigo topo (que pode ser NULL
    // se a pilha estava vazia — sem caso especial!).
    novo->proximo = p->topo;
    p->topo = novo;
}

Note: nenhum if para diferenciar pilha vazia ou não.

pilha_desempilhar

int pilha_desempilhar(Pilha* p) {
    if (pilha_vazia(p)) {
        printf("erro: pilha vazia\n");
        exit(1);
    }
    No* removido = p->topo;
    int valor = removido->valor;

    // O topo "afunda" um nivel. Se removido era o ultimo,
    // removido->proximo era NULL e topo vira NULL — sem caso especial.
    p->topo = removido->proximo;
    free(removido);
    return valor;
}
Eddy

Turma, atenção à ordem em desempilhar: primeiro salvamos removido->proximo em topo, e só depois chamamos free. Inverter essa ordem deixa topo apontando para memória que acabou de ser liberada — um ponteiro suspenso que provoca falhas obscuras na próxima operação.

topo, vazia, destruir

int pilha_topo(Pilha* p) {
    if (pilha_vazia(p)) {
        printf("erro: pilha vazia\n");
        exit(1);
    }
    return p->topo->valor;
}

int pilha_vazia(Pilha* p) {
    return p->topo == NULL;
}

void pilha_destruir(Pilha* p) {
    No* atual = p->topo;
    while (atual != NULL) {
        No* prox = atual->proximo;
        free(atual);
        atual = prox;
    }
    free(p);
}

main() — programa demonstrativo

int main(void) {
    Pilha* p = pilha_criar();
    pilha_empilhar(p, 10);
    pilha_empilhar(p, 20);
    pilha_empilhar(p, 30);

    printf("Topo da pilha: %d\n", pilha_topo(p));

    printf("Desempilhando: ");
    while (!pilha_vazia(p)) {
        printf("%d ", pilha_desempilhar(p));
    }
    printf("\n");

    pilha_destruir(p);
    return 0;
}

A main() fica no mesmo pilha.c, ao final do arquivo.

Compilando e contraste com a Fila

gcc -Wall -Wextra -o pilha_demo pilha.c
./pilha_demo

Saída:

Topo da pilha: 30
Desempilhando: 30 20 10
Compare com a Fila: 10 20 30. Mesma representação (lista encadeada), resultados opostos — porque o contrato é diferente. O comportamento observável é definido pelo TAD, não pela estrutura.

Bloco 6

Exercícios práticos

Exercício 1 — trace na mão

Pilha vazia. Sequência: emp(7), emp(3), emp(11), desemp, emp(5), topo, desemp, desemp.

Estado (topo → base) e retorno por operação.

Resposta:
  • emp(7) → [7]
  • emp(3) → [3, 7]
  • emp(11) → [11, 3, 7]
  • desemp → [3, 7] · retorna 11
  • emp(5) → [5, 3, 7]
  • topo → [5, 3, 7] · retorna 5
  • desemp → [3, 7] · retorna 5
  • desemp → [7] · retorna 3

Exercício 2 — pilha_imprimir

Imprime [30, 20, 10] do topo para a base, ou [] se vazia. Sem usar desempilhar.

Resposta: 
void pilha_imprimir(const Pilha* p) {
    printf("[");
    for (No* a = p->topo; a != NULL; a = a->proximo) {
        printf("%d", a->valor);
        if (a->proximo != NULL) printf(", ");
    }
    printf("]\n");
}
